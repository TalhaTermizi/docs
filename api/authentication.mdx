The eInvoices API uses API keys to authenticate every request. You include your key as a Bearer token in the `Authorization` HTTP header — no OAuth flows, no session cookies. This makes the API straightforward to call from any language, script, or tool that can make an HTTPS request. Keep your API keys private; they carry the same privileges as your account credentials.

## Generating an API Key

To generate a new API key, open your eInvoices dashboard and navigate to **Settings → API → New API Key**. Give the key a descriptive name (for example, `production-backend` or `billing-automation`) so you can identify it later. Choose the key type and permission level, then click **Create**. Copy the key immediately — for security reasons, eInvoices displays the full key value only once at creation time.

<Warning>
  eInvoices cannot show you the full key value again after you leave the
  creation screen. If you lose a key, revoke it and generate a new one.
</Warning>

## Live Keys vs. Test Keys

eInvoices issues two types of API keys so you can develop and test without affecting real data:

| Key type   | Prefix     | Behavior                                                                 |
| ---------- | ---------- | ------------------------------------------------------------------------ |
| **Live**   | `live_`    | Operates on real data. Sends invoice emails and triggers payment charges. |
| **Test**   | `test_`    | Operates on isolated test data. Does **not** send emails or charge cards. |

Use test keys throughout development and CI. Switch to a live key only when you are ready to go to production. Test and live data are completely separate — resources created with a test key are invisible to live-key requests and vice versa.

<Note>
  Treat test keys with the same care as live keys. A test key still grants
  access to your test environment data and counts toward your rate limits.
</Note>

## How to Authenticate

Pass your API key in the `Authorization` request header using the Bearer scheme:

```
Authorization: Bearer live_sk_your_key_here
```

The examples below show how to make an authenticated request to list invoices in three common environments:

<CodeGroup>

```bash curl
curl https://api.einvoices.com/v1/invoices \
  -H "Authorization: Bearer live_sk_your_key_here"
```

```python Python
import requests

api_key = "live_sk_your_key_here"
headers = {"Authorization": f"Bearer {api_key}"}

response = requests.get(
    "https://api.einvoices.com/v1/invoices",
    headers=headers
)
print(response.json())
```

```javascript Node.js
const response = await fetch('https://api.einvoices.com/v1/invoices', {
  headers: {
    'Authorization': 'Bearer live_sk_your_key_here'
  }
});
const data = await response.json();
```

</CodeGroup>

If the `Authorization` header is missing or the key is invalid, the API returns a `401 Unauthorized` response with an `invalid_api_key` or `authentication_required` error code. See the [Errors](/api/errors) page for the full list of authentication error codes.

## Key Permissions

When you create a key, choose one of two permission levels:

| Permission level | Can read data | Can create, update, delete |
| ---------------- | :-----------: | :------------------------: |
| **Read-only**    | ✓             | ✗                          |
| **Read-write**   | ✓             | ✓                          |

Use read-only keys for dashboards, reporting tools, or any integration that only needs to retrieve data. Use read-write keys for integrations that create or modify invoices, clients, payments, or webhooks. Calling a write endpoint with a read-only key returns `403 Forbidden` with the code `insufficient_permissions`.

<Tip>
  Follow the principle of least privilege — issue each integration the most
  restrictive permission level that still allows it to work correctly. This
  limits the blast radius if a key is ever compromised.
</Tip>

## Rotating and Revoking Keys

To rotate a key, generate a new one with the same name and permission level, update your integration to use the new key, verify that your integration is working correctly, and then revoke the old key from **Settings → API**. Revoking a key is immediate and permanent — any request that uses the revoked key returns `401 Unauthorized` right away.

Perform key rotations during a low-traffic window if possible, and monitor your error rates after the switch to catch any stale references to the old key.

## Security Best Practices

Follow these practices to keep your API keys secure:

- **Never commit keys to source control.** Even in private repositories, hardcoded keys are a significant risk. Use environment variables or a secrets manager instead.
- **Use environment variables.** Store your key in an environment variable such as `EINVOICES_API_KEY` and read it at runtime. Avoid logging or printing the value.
- **Rotate keys if compromised.** If you suspect a key has been exposed — for example, in a public commit or a shared log file — revoke it immediately and issue a replacement.
- **Use separate keys per environment.** Keep distinct keys for development, staging, and production so that a compromised development key cannot affect real data.
- **Audit your keys regularly.** Review the key list in **Settings → API** periodically and revoke any keys that are no longer in use.

<Warning>
  If you accidentally commit an API key to a public repository, revoke it
  immediately — do not wait until you have pushed a fix. Bots scan public
  commits for leaked credentials within seconds of a push.
</Warning>
