Webhooks let your application react to events in eInvoices in real time — without polling. When something happens in your account (an invoice is paid, a payment fails, a new client is created), eInvoices sends an HTTP `POST` request to the URL you register, carrying a JSON payload that describes the event. You can use webhooks to trigger downstream workflows like updating your CRM, sending Slack notifications, or reconciling payments in your accounting system.

## Base URL

```
https://api.einvoices.com/v1
```

## Authentication

Include your API key as a Bearer token in the `Authorization` header of every request.

```http
Authorization: Bearer YOUR_API_KEY
```

---

## Supported Events

eInvoices fires a webhook for each of the following event types. Subscribe to the specific events your application needs — you do not need to subscribe to all of them.

| Event | Description |
|---|---|
| `invoice.created` | A new invoice was created. |
| `invoice.sent` | An invoice was emailed to a client. |
| `invoice.viewed` | A client opened the invoice in their browser. |
| `invoice.paid` | An invoice was fully paid. |
| `invoice.overdue` | An invoice passed its due date without being fully paid. |
| `invoice.voided` | An invoice was voided. |
| `payment.completed` | A payment was successfully processed or recorded. |
| `payment.failed` | A payment attempt through the gateway failed. |
| `payment.refunded` | A payment was refunded. |
| `client.created` | A new client was created. |

## Webhook Payload Structure

Every webhook delivery uses the same top-level envelope regardless of event type.

<ResponseField name="id" type="string">
  Unique identifier for the event, prefixed with `evt_`. Use this to deduplicate deliveries — eInvoices may deliver the same event more than once.
</ResponseField>

<ResponseField name="type" type="string">
  The event type (e.g. `invoice.paid`). Use this field to route the payload to the correct handler in your application.
</ResponseField>

<ResponseField name="created_at" type="string">
  ISO 8601 timestamp of when the event occurred.
</ResponseField>

<ResponseField name="data" type="object">
  Contains the resource object relevant to the event. The key inside `data` matches the resource type (e.g. `invoice`, `payment`, `client`).
</ResponseField>

### Example Payload: `invoice.paid`

```json
{
  "id": "evt_abc123",
  "type": "invoice.paid",
  "created_at": "2024-06-15T14:31:00Z",
  "data": {
    "invoice": {
      "id": "inv_a1b2c3d4",
      "number": "INV-0042",
      "status": "paid",
      "total": 1050.00,
      "amount_paid": 1050.00
    }
  }
}
```

<Note>
  The `data` object contains a snapshot of the resource at the time of the event. For the full resource details, use the `id` to fetch the object via the corresponding REST endpoint.
</Note>

## Registering a Webhook Endpoint

### POST /v1/webhooks

Register a new webhook endpoint to start receiving event notifications. Your endpoint must be publicly reachable and must use HTTPS.

#### Request Body

<ParamField body="url" type="string" required>
  The HTTPS URL that eInvoices will POST event payloads to. HTTP URLs are rejected.
</ParamField>

<ParamField body="events" type="array" required>
  An array of event type strings you want to subscribe to (e.g. `["invoice.paid", "payment.failed"]`). Pass `["*"]` to subscribe to all current and future events.
</ParamField>

<ParamField body="secret" type="string">
  A secret string you choose (recommended: at least 32 random characters). eInvoices uses this to sign each payload so you can verify the request came from eInvoices. See [Signature Verification](#signature-verification) below.
</ParamField>

#### Example Request

```bash
curl -X POST https://api.einvoices.com/v1/webhooks \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://yourapp.com/webhooks/einvoices",
    "events": ["invoice.paid", "invoice.overdue", "payment.failed"],
    "secret": "whsec_your_random_secret_here"
  }'
```

#### Example Response

```json
{
  "id": "wh_m1n2o3p4",
  "url": "https://yourapp.com/webhooks/einvoices",
  "events": ["invoice.paid", "invoice.overdue", "payment.failed"],
  "created_at": "2024-06-01T08:00:00Z"
}
```

## DELETE /v1/webhooks/{id}

Delete a webhook endpoint. eInvoices immediately stops sending events to the associated URL. In-flight deliveries that have already begun may still be attempted.

### Path Parameters

<ParamField path="id" type="string" required>
  The unique ID of the webhook endpoint to delete (e.g. `wh_m1n2o3p4`).
</ParamField>

### Example Request

```bash
curl -X DELETE https://api.einvoices.com/v1/webhooks/wh_m1n2o3p4 \
  -H "Authorization: Bearer YOUR_API_KEY"
```

### Example Response

A successful deletion returns `204 No Content` with an empty body.

## Signature Verification

eInvoices signs every webhook payload using HMAC-SHA256 with the secret you provided when registering the endpoint. The signature is sent in the `X-eInvoices-Signature` request header in the format `sha256=<hex_digest>`.

You should **always verify this signature** before processing a webhook payload. This confirms the request genuinely came from eInvoices and that the payload was not tampered with in transit.

**How to verify:**
1. Read the raw request body as bytes — do not parse it as JSON first.
2. Compute an HMAC-SHA256 digest of the raw body using your webhook secret as the key.
3. Prepend `sha256=` to the hex-encoded digest.
4. Compare your computed value to the `X-eInvoices-Signature` header using a constant-time comparison function to prevent timing attacks.

<Warning>
  Always use a constant-time comparison (such as `crypto.timingSafeEqual` in Node.js or `hmac.compare_digest` in Python) when checking the signature. Standard string equality (`===`, `==`) is vulnerable to timing attacks.
</Warning>

<CodeGroup>

```javascript Node.js
const crypto = require('crypto');

function verifySignature(payload, signature, secret) {
  const expected = 'sha256=' + crypto
    .createHmac('sha256', secret)
    .update(payload)
    .digest('hex');

  // Use constant-time comparison to prevent timing attacks
  return crypto.timingSafeEqual(
    Buffer.from(signature),
    Buffer.from(expected)
  );
}

// Express.js example — use raw body middleware
app.post('/webhooks/einvoices', express.raw({ type: 'application/json' }), (req, res) => {
  const signature = req.headers['x-einvoices-signature'];

  if (!verifySignature(req.body, signature, process.env.WEBHOOK_SECRET)) {
    return res.status(401).send('Invalid signature');
  }

  const event = JSON.parse(req.body);

  switch (event.type) {
    case 'invoice.paid':
      handleInvoicePaid(event.data.invoice);
      break;
    case 'payment.failed':
      handlePaymentFailed(event.data.payment);
      break;
    // handle other event types...
  }

  res.status(200).send('OK');
});
```

```python Python
import hmac
import hashlib
from flask import Flask, request, abort

app = Flask(__name__)

def verify_signature(payload: bytes, signature: str, secret: str) -> bool:
    expected = 'sha256=' + hmac.new(
        secret.encode(),
        payload,
        hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(signature, expected)

@app.route('/webhooks/einvoices', methods=['POST'])
def handle_webhook():
    signature = request.headers.get('X-eInvoices-Signature', '')
    payload = request.get_data()  # raw bytes — do not call request.json yet

    if not verify_signature(payload, signature, WEBHOOK_SECRET):
        abort(401, 'Invalid signature')

    event = request.get_json()

    if event['type'] == 'invoice.paid':
        handle_invoice_paid(event['data']['invoice'])
    elif event['type'] == 'payment.failed':
        handle_payment_failed(event['data']['payment'])
    # handle other event types...

    return '', 200
```

</CodeGroup>

## Retry Policy

Your webhook endpoint must return a `2xx` HTTP status code within **30 seconds** of receiving the request. If it does not — whether due to a timeout, a non-2xx response code, or a connection error — eInvoices marks the delivery as failed and retries automatically.

eInvoices retries failed deliveries up to **5 times** using exponential backoff:

| Attempt | Delay after previous failure |
|---|---|
| 1st retry | 1 minute |
| 2nd retry | 5 minutes |
| 3rd retry | 30 minutes |
| 4th retry | 2 hours |
| 5th retry | 8 hours |

If all 5 retries fail, the delivery is marked permanently failed and no further attempts are made. You can view failed deliveries in the eInvoices dashboard under **Settings → Webhooks**.

<Note>
  Because retries can cause the same event to be delivered more than once, make your webhook handler **idempotent**. Use the `id` field on the event payload to detect and safely ignore duplicate deliveries.
</Note>

## Best Practices

- **Respond quickly.** Return a `200 OK` as soon as you receive and verify the payload, then process the event asynchronously in a background job. Slow handlers risk timing out and triggering unnecessary retries.
- **Log raw payloads.** Store the raw JSON payload and event `id` before processing. This makes it easy to replay events during debugging.
- **Handle unknown event types gracefully.** Your handler should not throw an error on event types it does not recognise — eInvoices may introduce new event types in the future.
- **Rotate your secret periodically.** Update your webhook registration with a new secret via `PUT /v1/webhooks/{id}` and deploy the corresponding change to your verification logic simultaneously.
