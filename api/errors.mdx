When something goes wrong, the eInvoices API returns a standard HTTP status code alongside a JSON error body that tells you exactly what failed and why. Design your integration to inspect the error response rather than relying solely on the HTTP status code — the error codes are stable, machine-readable identifiers that let you write precise error-handling logic for each failure scenario.

## Error Response Format

Every error response has the same JSON structure: a top-level `error` object containing a `code`, a human-readable `message`, and an optional `details` array. The `details` array is present on validation errors and lists each field that failed along with a specific message.

```json
{
  "error": {
    "code": "validation_error",
    "message": "The request body contains invalid fields.",
    "details": [
      { "field": "due_date", "message": "must be after issue_date" }
    ]
  }
}
```

| Field           | Type            | Description                                                                 |
| --------------- | --------------- | --------------------------------------------------------------------------- |
| `error.code`    | string          | A stable, machine-readable error identifier. Use this in your error-handling logic. |
| `error.message` | string          | A human-readable description of the error. Useful for logging and debugging, but not guaranteed to remain unchanged between releases. |
| `error.details` | array (optional)| Present on validation errors. Each item contains a `field` name and a field-level `message`. |

<Note>
  Always branch on `error.code` in your application code, not on
  `error.message`. The message is intended for human readers and may be revised
  over time, while the code is a stable contract.
</Note>

## HTTP Status Codes

The eInvoices API uses the following standard HTTP status codes:

| Status                        | Meaning                                                               |
| ----------------------------- | --------------------------------------------------------------------- |
| `200 OK`                      | The request succeeded. The response body contains the requested data. |
| `201 Created`                 | A new resource was successfully created. The response body contains the created object. |
| `400 Bad Request`             | The request was malformed or contained invalid parameters.            |
| `401 Unauthorized`            | The API key is missing or invalid.                                    |
| `403 Forbidden`               | The API key is valid but lacks permission to perform the action.      |
| `404 Not Found`               | The requested resource does not exist.                                |
| `409 Conflict`                | The request conflicts with the current state — for example, a duplicate invoice number. |
| `422 Unprocessable Entity`    | The request was well-formed but failed field-level validation.        |
| `429 Too Many Requests`       | You have exceeded the rate limit. Check the `Retry-After` header.    |
| `500 Internal Server Error`   | An unexpected error occurred on the eInvoices server.                |
| `502 Bad Gateway`             | The payment gateway returned an upstream error. Retry after a short delay. |

<Tip>
  Use the HTTP status code for coarse-grained branching (for example, retry on
  `500`, surface a user message on `422`) and the `error.code` for fine-grained
  logic (for example, handle `invoice_already_sent` specifically).
</Tip>

## Error Codes

The following error codes can appear in the `error.code` field:

| Code                        | HTTP Status | Description                                                           |
| --------------------------- | ----------- | --------------------------------------------------------------------- |
| `authentication_required`   | 401         | No API key was provided in the `Authorization` header.                |
| `invalid_api_key`           | 401         | The API key is malformed, expired, or has been revoked.               |
| `insufficient_permissions`  | 403         | The API key is read-only and was used on a write endpoint.            |
| `resource_not_found`        | 404         | The resource ID in the request URL does not correspond to any existing record. |
| `validation_error`          | 422         | One or more fields in the request body failed validation. Check `error.details` for field-level messages. |
| `duplicate_invoice_number`  | 409         | An invoice with the same invoice number already exists in your account. |
| `invoice_already_sent`      | 409         | The invoice has already been sent to the client and can no longer be edited. |
| `rate_limit_exceeded`       | 429         | You have sent too many requests. Wait for the number of seconds specified in the `Retry-After` header before retrying. |
| `gateway_error`             | 502         | The payment gateway returned an error when processing the request.    |

## Handling Errors in Your Integration

The examples below show how to handle errors defensively, inspecting `error.code` to distinguish between different failure types and responding appropriately to each.

<CodeGroup>

```python Python
import requests
import time

def get_invoices(api_key: str) -> dict:
    url = "https://api.einvoices.com/v1/invoices"
    headers = {"Authorization": f"Bearer {api_key}"}

    response = requests.get(url, headers=headers)

    if response.ok:
        return response.json()

    error = response.json().get("error", {})
    code = error.get("code")
    message = error.get("message", "Unknown error")

    if code == "authentication_required" or code == "invalid_api_key":
        raise PermissionError(f"Authentication failed: {message}")

    elif code == "insufficient_permissions":
        raise PermissionError(f"API key lacks permission: {message}")

    elif code == "rate_limit_exceeded":
        retry_after = int(response.headers.get("Retry-After", 5))
        print(f"Rate limit hit. Retrying in {retry_after}s...")
        time.sleep(retry_after)
        return get_invoices(api_key)  # Retry once

    elif code == "validation_error":
        details = error.get("details", [])
        for detail in details:
            print(f"  Field '{detail['field']}': {detail['message']}")
        raise ValueError(f"Validation failed: {message}")

    else:
        raise RuntimeError(f"API error [{code}]: {message}")
```

```javascript Node.js
async function getInvoices(apiKey) {
  const response = await fetch('https://api.einvoices.com/v1/invoices', {
    headers: { 'Authorization': `Bearer ${apiKey}` }
  });

  if (response.ok) {
    return response.json();
  }

  const { error } = await response.json();

  switch (error.code) {
    case 'authentication_required':
    case 'invalid_api_key':
      throw new Error(`Authentication failed: ${error.message}`);

    case 'insufficient_permissions':
      throw new Error(`API key lacks permission: ${error.message}`);

    case 'rate_limit_exceeded': {
      const retryAfter = parseInt(response.headers.get('Retry-After') ?? '5', 10);
      console.warn(`Rate limit hit. Retrying in ${retryAfter}s...`);
      await new Promise(resolve => setTimeout(resolve, retryAfter * 1000));
      return getInvoices(apiKey); // Retry once
    }

    case 'validation_error':
      error.details?.forEach(({ field, message }) => {
        console.error(`  Field '${field}': ${message}`);
      });
      throw new Error(`Validation failed: ${error.message}`);

    default:
      throw new Error(`API error [${error.code}]: ${error.message}`);
  }
}
```

</CodeGroup>

## Rate Limit Errors

When you hit the rate limit, the response includes a `Retry-After` header alongside the `429` status code. Always read this header rather than using a fixed backoff interval — the value reflects the exact number of seconds until your limit resets.

```
HTTP/1.1 429 Too Many Requests
Retry-After: 8
Content-Type: application/json

{
  "error": {
    "code": "rate_limit_exceeded",
    "message": "Rate limit exceeded. Please wait before retrying."
  }
}
```

<Tip>
  For high-throughput integrations, implement a request queue with a token
  bucket or leaky bucket algorithm to spread requests evenly and avoid hitting
  the limit in the first place.
</Tip>

## Server Errors

A `500 Internal Server Error` indicates an unexpected failure on the eInvoices side. These errors are rare but can occur during infrastructure incidents. Retry `500` responses with exponential backoff — most transient server errors resolve within a few seconds. If a `500` persists for more than a few minutes, check the [eInvoices status page](https://status.einvoices.com) or contact [support@einvoices.com](mailto:support@einvoices.com).

<Warning>
  Do not retry `4xx` errors automatically (except `429`). A `400`, `422`, or
  `409` response means the request itself is the problem — retrying the same
  request without changing it will produce the same error.
</Warning>
